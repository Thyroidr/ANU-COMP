Measuring Algorithmic Efficiency - Time Complexity

Posted on 2021-01-23 In Knowledge

Based on Introduction to Algorithms, 3rd Edition, Chapter 3
By Ryan Wang

About

It is often crucial to understand the efficiency of an algorithm. For example, we may have algorithms $A$ and $B$ to solve a problem. $A$ may slow but easy to implement and its implementations are likely bug-free, while $B$ is fast yet extremely difficult to implement. Knowing an exact measure of their efficiencies can help determining which one to use in different situations. In this example, the we have low demand for efficiency (small data sets), $A$ would be preferred, while $B$ would be ideal with larger data sets.

We need a quantitative measurement of different situations, and that is often the size of data that we wish to process. This "size of data" is called $n$: it can refer to the number of items in knap-sack, number of data entries in excel, etc.

Time Complexity

Time complexity is notation to describe how efficient an algorithm is in terms of its running time, that is, how long it would run on a computer. Depending on the circumstance, we may use one of three different types of time complexity: upperbound, asymptotic, lowerbound. In each of these cases, we use a function $f(n)$ inside a one of $O()$, $\Theta()$, and $\Omega()$.

Notation Description
$O(f(n))$ Upperbound
$\Theta(f(n))$ Asymptotic
$\Omega(f(n))$ Lowerbound

The interior function

I will get into what "upperbound", "asymptotic", and "lowerbound" mean in a moment. But first, we will look at $f(n)$. These notations use the function $f(n)$ to describe the rate of growth of algorithmic runtimes. Exact algorithmic runtimes vary significantly from machine to machine, and correspondingly time complexities can only be indicative. Due to this indicative nature, the function $f(n)$ is nearly always simplified by omitting constant factors and minor terms. For example, if we know that the runtime on a machine is exactly

$$f(n) = 5n^2 + 3n + 2$$

This is generalised to most machines by simplifying to

$$f(n) = n^2$$

In the example above, we know definitively that the $n^2$ term grows faster than the other terms starting from very small $n$. Thus for all $n$ we really care about, it is the dominant term. We may have multiple dominant terms, especially (although not always) in algorithms where the time complexity is determined by multiple variables. One such example is with the infamous Dijkstra's algorithm for shortest paths in Graph Theory. This algorithm has its run time determined by both the number of vertices ($|V|$) and edges ($|E|$) in the graph. Its time complexity, when implemented using a priority queue for optimisation, is
$$O(|E| + |V| \log |V|)$$

The three types of time complexity

Now onto what the $O()$, $\Theta()$ and $\Omega()$ notations mean. Well, they mean what you'd expect. $O()$ gives the hard cap on the time complexity, and that an algo with $O(n^2)$ would never run higher than $n^2$ time complexity, although it could run with $n$ time complexity in certain cases. $\Omega()$ is pretty much the same except with lowerbound, while $\Theta()$ tells you both the upper and lower bounds.

There is one catch though: the runtime only has to obey these bounds after some constant $c$. This constant $c$ can be any number. And thus a function running in $\Theta(n^3)$ for $n \le 1000$ and $\Theta(n^2)$ for $n > 1000$ can still be said to have time complexity $\Theta(n^2)$. This is a rare example, and hardly occurs unless if you "Frankenstein" two algorithms together and have them solve for different sizes of input (I wonder why you would do that hmmm).

Once again, time complexities are only indicative, optimising a piece of code does not necessarily involve optimising its time complexity. Constant factor optimisation (improving runtime but do not changing time complexity) is in fact quite common.
